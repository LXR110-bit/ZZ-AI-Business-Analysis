# 机型/品类周数据 Base 分片规则方案

> **日期**: 2026-07-07  
> **性质**: Base 迁移分片契约方案，供历史回刷和每日增量写入共同遵守  
> **背景**: 用户确认当前飞书多维表格单表限制为 50,000 行；Base 导入不能再按任意行号粗切，必须用可解释、可检索、可复现的业务分片规则。

---

## 1. 目标

1. 每张物理 Base 数据表行数必须小于等于安全阈值：`49,000` 行。
2. 分片规则必须稳定、可解释，用户能通过月份、周、维度、品类范围找到数据。
3. 每日增量写入必须复用同一规则，不能历史回刷一套、日常 cron 一套。
4. 下游只能从 `周索引.active=true` 读取当前版本，不能直接猜表名。
5. 旧版本保留回滚，但不参与下游统计。

---

## 2. 核心术语

| 术语 | 含义 |
|---|---|
| Base | 用户按月份、表类型预建的飞书多维表格，例如 `机型维度汇总6月` |
| 逻辑表 | 业务维度表，例如 `日期机型`、`估价成色`、`履约`、`估价成色履约`、`质检成交` |
| 物理表 | Base 内真实导入的数据表，一个逻辑表可以被拆成多个物理表 |
| 分片 | 一个物理表对应的一段业务数据，必须能通过分片规则定位 |
| active 版本 | `周索引.active=true` 的最新可读版本 |
| run_id | 一次导入/发布的版本号 |

---

## 3. 安全阈值

- 平台限制：单 Base 表 50,000 行。
- 实施阈值：`BASE_PARTITION_ROW_LIMIT=49000`。
- 原因：预留 1,000 行安全边界，避免边界值、表头/系统行、平台口径差异导致导入或 UI 失败。

任何导入包在发布前必须校验：

```text
forall physical_table: row_count <= 49000
sum(physical_table.rows for same logical table) == source_logical_table.rows
sum(metric_sums of partitions) == source_logical_table.metric_sums
```

---

## 4. 分片优先级

分片顺序固定如下，不能按原始行号随意切：

### 4.1 第一层：按统计周拆

从 `日期` 字段计算 ISO 周，格式为 `YYYY-Www`。

- W18 = 2026-04-27～2026-05-03，归属 4 月 Base。
- W19-W22 归属 5 月 Base。
- W23-W27 归属 6 月 Base。
- 后续仍按“周一所在月”归属月份。

如果某个逻辑表的某个周切片 `<= 49,000` 行，则直接生成 1 张物理表。

表名格式：

```text
{MM}_{Www}_{汇总|日均}_{逻辑维度别名}
```

示例：

```text
06_W23_汇总_日期机型
06_W24_日均_履约
05_W19_汇总_质检成交
```

### 4.2 第二层：按品类名称分桶

如果某个“统计周 + 逻辑表”仍然超过 49,000 行，则按 `品类名称` 分桶。

规则：

1. 先按 `品类名称` 聚合行数。
2. 按标准化后的 `品类名称` 升序排列。
3. 从小到大贪心装桶：加入下一个品类后若超过 49,000 行，则新开一个桶。
4. 一个品类不能被拆到多个桶，除非单一品类本身超过 49,000 行。
5. 每个桶必须记录品类范围和品类列表摘要，供用户检索。

表名格式：

```text
{MM}_{Www}_{汇总|日均}_{逻辑维度别名}_C{nn}
```

示例：

```text
06_W23_汇总_估价成色_C01
06_W23_汇总_估价成色_C02
06_W23_汇总_估价成色_C03
```

### 4.3 第三层：单品类超限兜底

如果某个 `统计周 + 逻辑表 + 品类名称` 本身超过 49,000 行，才允许使用兜底规则。

兜底规则：

1. 按稳定业务键排序：`机型ID`、`机型名称`、其余维度字段。
2. 生成 `R01/R02/...` 行内分片。
3. `周索引` 必须记录 `fallback_reason=single_category_over_limit`。
4. 通知里必须提示人工关注，因为这说明该品类体量已经超过当前 Base 单表承载能力。

表名格式：

```text
{MM}_{Www}_{汇总|日均}_{逻辑维度别名}_C{nn}_R{nn}
```

当前 2026-04～2026-06 实测不需要第三层兜底：`估价成色` 最大单周单品类约 48,422 行，仍低于 49,000。

## 6. 索引表契约

现有 `周索引` 继续作为下游入口，但需要扩展分片字段。

### 6.1 必备字段

| 字段 | 说明 |
|---|---|
| 记录键 | `{统计周}|{表类型}|{source_sheet_id}|{partition_key}|{run_id}` |
| 逻辑表键 | `{统计周}|{表类型}|{source_sheet_id}` |
| 数据月份 | `YYYY-MM` |
| 统计周 | `YYYY-Www` |
| run_id | 本次导入版本 |
| active | 是否当前可用版本 |
| 表类型 | `summary` / `daily_avg` |
| 业务表名 | 中文业务维度名 |
| Base表名 | 物理表名 |
| Base表ID | 导入后的 table_id |
| 行数 | 物理表行数，必须 <= 49,000 |
| 列数 | 物理表列数 |
| 分片规则版本 | 例如 `base_partition_v1` |
| 分片字段 | `none` / `统计周` / `品类名称` / `fallback_row_range` |
| 分片序号 | `C01` / `R01` 等 |
| 分片总数 | 同一逻辑表、同一周下的分片数 |
| 品类范围 | 该桶内品类名最小值～最大值 |
| 品类数 | 该桶包含的品类数量 |
| 品类列表摘要 | 品类列表的 hash 或短摘要 |
| 校验结果 | `manifest_matched` |
| 状态 | `已发布` / `已归档` |
| 导入时间 | 发布时间 |
| 备注 | 导入方式、兜底原因等 |

### 6.2 下游读取规则

下游不得直接按表名猜数据表，必须：

1. 读取目标 Base 的 `周索引`。
2. 筛选 `active=true`。
3. 按 `数据月份 + 统计周 + 表类型 + source_sheet_id` 找到全部物理分片。
4. 对同一逻辑表的多个物理表做 union/concat。
5. 若用户查询指定品类，可先在索引里用 `品类范围/品类列表摘要` 定位候选表，再进表内筛选。

## 7. 每日增量写入规则

每日 cron 不做“追加到旧表”的隐式写入，统一走版本发布：

1. 计算本次受影响的统计周：通常是当前周；若 SQL 回补历史，则包含被回补的周。
2. 对每个受影响周，重新生成该周对应逻辑表的完整数据。
3. 按本方案分片生成新的物理表。
4. 导入完成后做校验：
   - 物理表行数全部 <= 49,000。
   - 分片总行数等于源逻辑表行数。
   - 核心指标合计等于源逻辑表合计。
   - 每行数据只属于一个分片。
5. 校验通过后发布 `周索引`：
   - 新 run 的分片 `active=true`。
   - 同一 `逻辑表键` 的旧 active 记录改为 `active=false`、`状态=已归档`。
6. 通知消息同时给出：Base 链接、run_id、受影响周、分片数量、行数校验结果、旧 Sheets 回滚链接。

这样每日数据变化不会污染旧表，也不会因为追加导致某张表悄悄超过 50,000 行。

## 8. 历史回刷修正方案

当前 2026-04～2026-06 机型数据需要按本规则重新发布一个新版本：

```text
run_id = hist_20260427_20260705_v2_49k
partition_rule_version = base_partition_v1
row_limit = 49000
```

品类数据不需要重跑，因为当前每张物理表都远低于 50,000 行。

### 8.1 预计物理表数量

| Base | 当前逻辑表数 | 新规则物理表数 |
|---|---:|---:|
| 机型汇总4月 | 5 | 7 |
| 机型日均4月 | 5 | 7 |
| 机型汇总5月 | 5 | 30 |
| 机型日均5月 | 5 | 30 |
| 机型汇总6月 | 5 | 39 |
| 机型日均6月 | 5 | 39 |

### 8.2 实测依据

- 4 月只有 W18。仅 `估价成色` 超过 49,000，需要按品类桶拆为 3 张表。
- 5 月 W19-W22。除 `估价成色` 外，其余逻辑表按周拆即可；`估价成色` 还需按品类桶拆。
- 6 月 W23-W27。同 5 月，但包含 5 个完整周，因此物理表更多。
- 2026-04～2026-06 实测最大 `统计周 + 品类名称` 行数低于 49,000，不需要第三层兜底。

## 9. 代码落地范围

### 9.1 配置

新增常量或配置：

```python
BASE_PARTITION_ROW_LIMIT = 49000
BASE_PARTITION_RULE_VERSION = "base_partition_v1"
```

### 9.2 新增模块/函数建议

在 `skills/workflows/机型周数据/base_migration.py` 内增加或拆出：

| 函数 | 作用 |
|---|---|
| `assign_iso_week(date)` | 由日期计算 ISO 周 |
| `build_partition_plan(df, table_meta, row_limit)` | 生成分片计划 |
| `partition_by_week(df)` | 第一层按周拆 |
| `partition_by_category_bucket(df, row_limit)` | 第二层按品类桶拆 |
| `validate_partition_plan(plan)` | 校验每个分片行数和覆盖完整性 |
| `write_partitioned_base_package(...)` | 输出按分片命名的 xlsx + manifest |
| `publish_partition_manifest_to_index(...)` | 写扩展后的 `周索引` |

### 9.3 CLI 行为

新增可选参数：

```bash
--base-partition-rule base_partition_v1
--base-row-limit 49000
```

默认规则：

- Base migration import 默认启用 `base_partition_v1`。
- 如未指定 `--base-row-limit`，默认 49,000。
- 若任一分片超过限制，导入前 fail fast，不写 Base。

## 11. 风险与约束

| 风险 | 应对 |
|---|---|
| 物理表数量增加 | 通过命名规则和 `周索引/分片索引` 管理，不靠肉眼猜表 |
| 用户按表名找品类困难 | 索引记录品类范围、品类列表摘要，必要时增加单独 `分片索引` 表 |
| 后续数据增长导致单品类超限 | 第三层 fallback，并在通知中报警 |
| 旧大表占用 Base 空间 | 初期不删除，确认新版本稳定后再单独走删除方案 |
| 下游未按 active 索引读取 | 下游改造必须作为上线前置条件 |

---

## 12. 推荐决策

采用 `base_partition_v1`：

1. **周优先**。
2. **超限再按品类名称分桶**。
3. **严禁纯行号随机切分**。
4. **每日增量也重建受影响周并发布新 active 版本**，不做隐式追加。
5. **下游统一从 `周索引.active=true` 读取所有分片并合并**。

该规则同时满足：单表不超过 50,000 行、用户可按周和品类定位、每日 skill 可稳定复用。
