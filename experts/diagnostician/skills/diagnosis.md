# 异动诊断 Skill

> 本 Skill 是专家 C 的主战场。任何"为什么"类问题都强制走此流程。
> **不允许**只看"是什么变了"，必须把"为什么变 + 该做什么"答全。

---

## 一、输入识别

从用户问题里提取：
- **诊断对象**：哪个品类 / 渠道 / 人群 / 实验组
- **诊断指标**：成交量 / GMV / 转化率 / 客单价 / 复购率 / 留存
- **异动幅度**：用户感受的变化（"跌得厉害"≠"跌 5%"，需要先量化）
- **时间窗口**：本周 vs 上周？本周 vs 去年同期？连续 N 周下滑？
- **背景信息**：用户是否提到近期活动 / 改版 / 实验 / 外部事件

> 输入不明确就反问，**禁止瞎猜**。尤其要先确认"异动幅度"和"时间窗口"。

---

## 二、诊断流程（强制顺序）

### 步骤 1：量化异动（不量化不诊断）

1. 调 `read_email` + `parse_csv` 取原始数据
2. 调 `calc_caliber` 算同比 + 环比
3. 调 `query_metric` 校准口径（确保口径无歧义）
4. 调 `get_baseline`（如有）拿历史正常波动区间
5. 输出量化结论：`<指标> <周期> <变化幅度>，超出 <基线波动区间> X 倍`

**判定标准**：
- 变化幅度 < 历史 1 倍标准差 → 噪声，不诊断（说明"在正常波动内"即结案）
- 变化幅度 ≥ 1 倍标准差 → 进入步骤 2
- 变化幅度 ≥ 3 倍标准差 → 升级为 P0 异常

### 步骤 2：§4 异动诊断四问（**逐问回答，缺一不可**）

#### 问 1：数据可信吗？

排查三件事：
- **口径**：本周和对比周的口径是否一致？字段定义有没有变？（查 `query_metric` 历史版本）
- **采集**：埋点是否正常？是否有数据缺失 / 重复 / 异常值？
- **归因**：归因模型是否升级？归因窗口是否变化？

**输出**：
- ✓ 可信 / ✗ 不可信（说明原因）/ ⚠ 部分可信（说明哪部分需要打折）

> 跳过此问 = 后面所有结论作废。曾踩过的坑：数据 BI 把"成交订单"口径从"已签收"改成"已下单"，结果"跌 30%"是口径变化不是业务跌。

#### 问 2：短期波动还是结构性变化？

判断维度：
- **节假日**：本期 vs 对比期是否跨春节 / 国庆 / 618 / 双 11？跨节假日的同比必须用"对齐节假日"口径
- **活动**：本期 / 对比期是否有大促 / 补贴 / 改版？
- **持续性**：单周下滑 ≠ 趋势。看连续 3-4 周的曲线，找拐点

**输出**：
- 短期波动（具体原因）/ 结构性变化（连续 N 周下滑 + 趋势线）/ 待定（数据不足，需要观察）

#### 问 3：外部市场带的还是内部动作带的？

并列查两边：
- **外部**：
  - 行业大盘数据（爱回收 / 闲鱼 / 转转大盘）同期变化
  - 竞品价格走势（如有抓价数据）
  - 季节 / 用户认知 / 政策变化
- **内部**：
  - 我们的活动 / 价格 / 投放 / 改版日历
  - 产品 / 算法 / 运营动作时间线
  - 渠道流量来源结构变化

**输出**（必须给出比重）：
- 全外部：行业整体跌 X%，我们跌 Y%，差距在合理范围
- 全内部：行业平稳，我们独家变化
- 内外都有：拆出大致权重（如 "外部贡献 60% / 内部贡献 40%"）

> 这是诊断最容易偷懒的一问。**只查内部 = 诊断不完整**，必须主动找外部对照。

#### 问 4：如果是真的，对应什么动作？

接 §5 动作闭环原则。每个动作必须带：
| 字段 | 内容 |
|---|---|
| 动作描述 | 具体到执行单元，如 "iPhone 14 估价回调 2%" |
| 验证方法 | 用什么数据 / 实验确认有效 |
| 基线值 | 当前数据 |
| 预期增量 | 保守 / 中性 / 乐观三档 |
| 成本 | 人力 / 补贴 / 时间 |
| ROI 测算 | 增量收益 / 成本 |

**禁止输出**：
- 无动作的归因结论（= 无效观察，回炉）
- 无验证方法的动作（= 信口开河）
- 无 ROI 测算的补贴建议（= 不负责任）

### 步骤 3：§3 瓶颈定位（找真正的卡点）

异动通常出现在链路某一环，把全链条列出来：

```
曝光 → 进店 → 加购 → 估价 → 下单 → 成交 → 复购
```

1. 调 `split_dimension` 拆每一环的本期 vs 对比期数据
2. 找 **绝对值最异常** 或 **降幅最大** 的环节 = 瓶颈
3. 标注 vs 行业基线（如有 `get_baseline`）

**典型瓶颈识别**：
- 转化率 < 行业基线 50% → 主瓶颈
- 单一环节降幅 > 整体降幅 2 倍 → 主瓶颈
- 多环节均匀下滑 → 上游流量问题，非链路内问题

### 步骤 4：§1 三层穿透（每个结论都覆盖三层）

诊断结论必须同时回答：
- **上游侧**：新品 / 货源 / 上游平台是否变化？
- **市场侧**：行业 / 竞品 / 用户认知是否变化？
- **我们侧**：策略 / 转化 / 活动 / 算法是否变化？

**违反判定**：三层只答了 1-2 层 → 自检不通过，回炉。

## 四、自检清单（§6 强制）

输出前 Agent 内部勾选：

- [ ] 异动量化了吗？历史波动区间标了吗？
- [ ] 四问每问都答了吗？没有跳过的？
- [ ] 三层穿透三角度都覆盖了吗？
- [ ] 瓶颈定位有数据证据吗？
- [ ] 动作建议都带 §5 闭环吗？
- [ ] 一句话结论真的是一句话吗？（≤ 30 字）
- [ ] 所有数字都有来源吗？
- [ ] 没有未量化的形容词（"明显下滑"、"显著上升"）？

任一未达成 → **回炉重答**。

---

## 五、输出后操作

1. 调 `write_doc` 在飞书"诊断报告"文件夹下创建文档
2. 调 `send_im` 把文档链接 + 一句话结论推送给请求方
3. 如运营追问，转移到对话模式
4. 重要诊断（P0 级别）写入 `knowledge_base` 案例库（MVP-2 阶段）

## 六、常见陷阱（踩过的坑）

1. **口径变更未察觉**：BI 改字段定义后，"变化"其实是口径漂移。每次诊断都要 `query_metric` 验最新口径
2. **只看内部不查外部**：内部归因看起来很合理，但其实是行业大盘带的——必须主动查外部对照
3. **样本量太小下结论**：日活 < 100 的细分品类不下定性结论（§7 兜底）
4. **混用同比/环比**：写"同比 +5% 环比 -8%"时必须明确是哪个，不能跳着说
5. **建议无验证方法**：常见"加大投放"、"加强运营" → 必须细化到能量化验证
6. **跨春节同比未对齐**：1 月数据同比去年 1 月要看春节周对齐口径，否则结论失真
